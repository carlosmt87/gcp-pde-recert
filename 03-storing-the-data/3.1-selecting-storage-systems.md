## Key Concepts

> _Fill in your own notes here as you study._

- **BigQuery**: Serverless, columnar, petabyte-scale data warehouse. Optimized for
  analytical (OLAP) workloads. Supports SQL, streaming inserts, and federated queries.
  Use for: analytics, reporting, data warehouse, ML training data.

- **BigLake**: Unified storage engine that extends BigQuery's capabilities to data
  stored in Cloud Storage. Enables BigQuery-style access control and queries on external
  tables (Parquet, ORC, Avro). The "lakehouse" pattern on GCP.

- **AlloyDB**: Fully managed PostgreSQL-compatible database, optimized for demanding
  transactional and analytical workloads (HTAP). Higher performance than Cloud SQL for
  complex queries. Use for: enterprise OLTP workloads requiring PostgreSQL compatibility.

- **Bigtable**: Fully managed, wide-column NoSQL database. Sub-10ms latency at massive
  scale. Use for: time-series data, IoT telemetry, financial ticks, high-throughput
  reads/writes. Not suitable for complex joins or ad-hoc SQL.

- **Spanner**: Globally distributed, strongly consistent, relational database.
  Horizontal scaling with ACID transactions. Use for: global OLTP with high availability,
  financial systems, inventory management across regions.

- **Cloud SQL**: Managed relational database (MySQL, PostgreSQL, SQL Server).
  Regional, vertically scaled. Use for: standard OLTP workloads, web apps, CMS backends.
  Easier to migrate on-prem MySQL/PostgreSQL to.

- **Cloud Storage**: Object storage for unstructured data. Durability of 11 nines.
  Storage classes: Standard, Nearline (30-day), Coldline (90-day), Archive (365-day).
  Use for: data lake, staging, backups, media files.

- **Firestore**: Fully managed, serverless NoSQL document database. Real-time sync
  and offline support for mobile/web apps. Use for: user profiles, product catalogs,
  app state, real-time collaborative data.

- **Memorystore**: Managed Redis and Memcached. Sub-millisecond latency, in-memory.
  Use for: session caches, leaderboards, rate limiting, real-time counters.

## Practice Questions

### Question 1

A global e-commerce company processes 50,000 transactions per second across data centers
in the US, Europe, and Asia. Each transaction updates inventory in real-time and must
be immediately visible across all regions with strong consistency. Which storage service
should the data engineer recommend?

**A)** Cloud SQL with cross-region read replicas
**B)** Bigtable with multi-cluster routing
**C)** Cloud Spanner with a multi-region configuration
**D)** Firestore in Datastore mode

**Answer: C**

> Spanner provides horizontal scaling, global distribution, and strong consistency
> (external consistency — stronger than serializable). Multi-region Spanner handles
> 50K TPS across regions. Cloud SQL (A) read replicas are eventually consistent;
> Bigtable (B) doesn't provide relational ACID guarantees for inventory management.

---

### Question 2

A company collects sensor readings from 1 million IoT devices, generating 100,000
events per second. Each event contains a device ID, timestamp, and measurement value.
Analytics need to retrieve the last 24 hours of readings for any device in under 10ms.
Which storage service is most appropriate?

**A)** BigQuery with partitioned tables on timestamp
**B)** Cloud Spanner
**C)** Bigtable with a row key of `device_id#reverse_timestamp`
**D)** Cloud SQL with indexed timestamp column

**Answer: C**

> Bigtable is designed for time-series, high-throughput IoT data. The `device_id#reverse_timestamp`
> row key pattern enables efficient retrieval of the most recent readings for a device.
> BigQuery (A) is for analytics, not sub-10ms lookups. Cloud SQL (D) won't handle
> 100K events/sec without significant operational complexity.

### Question 3

A data team stores processed analytical data in BigQuery but also maintains a large
archive of raw data files (Parquet format) in Cloud Storage for cost reasons. They want
to run SQL queries that JOIN BigQuery tables with the Parquet files in Cloud Storage,
with BigQuery-level access controls applied to the Cloud Storage data. Which GCP feature
enables this?

**A)** BigQuery federated queries using Cloud Storage External Tables
**B)** BigLake tables on Cloud Storage with unified BigQuery access control
**C)** Cloud Storage FUSE mount with BigQuery connected sheets
**D)** Dataflow pipeline to copy Cloud Storage data to BigQuery before querying

**Answer: B**

> BigLake provides a unified metadata layer over Cloud Storage that enables BigQuery-
> style fine-grained access control (column-level, row-level) on external data. Unlike
> basic external tables (A), BigLake enforces IAM policies at the data level, not just
> the table level. No data movement needed (vs. D).

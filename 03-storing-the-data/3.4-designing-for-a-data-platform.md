## Key Concepts

> _Fill in your own notes here as you study._

- **Data platform vs. data lake**: A data platform is broader — it encompasses
  ingestion, storage, processing, governance, discovery, and consumption layers.
  A data lake is just one storage component. GCP's modern data platform often combines:
  Cloud Storage (lake) + BigQuery (warehouse) + Dataplex (governance) + Dataform (ELT)
  + Looker/Looker Studio (visualization).

- **Dataplex as the governance backbone**: Dataplex provides the central control plane
  for a data platform — managing metadata, data quality, access, lineage, and discovery
  across heterogeneous storage (Cloud Storage + BigQuery + AlloyDB).

- **Federated governance**: A governance model where central policies are defined but
  enforcement is distributed across domains/teams. Contrast with:
  - *Centralized*: One team controls all data — bottleneck at scale
  - *Federated*: Domain teams own their data; central team defines standards
  - Implemented via: Dataplex, org-level IAM policies, shared tagging taxonomy,
    Analytics Hub data sharing

- **Data Mesh on GCP**: An architectural pattern where data is treated as a product,
  owned by domain teams, with self-serve infrastructure. Dataplex supports data mesh
  by enabling per-domain lakes with central discoverability.

- **BigQuery as platform hub**: BigQuery serves as the primary query, storage, and
  analytics engine. Supports: multi-cloud (BigQuery Omni), lakehouse (BigLake),
  ML (BigQuery ML), and sharing (Analytics Hub).

- **Dataplex Catalog**: Manages technical and business metadata. Supports custom
  aspects (metadata schemas), tag templates, and search. Central to making data
  discoverable across a platform.

| Service | Role in Platform |
|---------|-----------------|
| **Dataplex** | Governance control plane — lakes, zones, data quality, lineage |
| **Dataplex Catalog** | Unified metadata and data discovery across all sources |
| **BigQuery** | Central analytical engine; data warehouse; ML; sharing |
| **Cloud Storage** | Data lake storage layer (raw and curated data) |
| **BigLake** | Lakehouse layer — BigQuery governance on Cloud Storage data |
| **Analytics Hub** | Data product publishing and cross-org/cross-project sharing |
| **Dataform** | ELT transformation layer within BigQuery |
| **Looker / Looker Studio** | Consumption layer — dashboards and data exploration |
| **IAM + Org Policy** | Org-wide policy enforcement for federated governance |

## Practice Questions

### Question 1

A large corporation has multiple business units (finance, marketing, operations), each
with their own data teams and GCP projects. They want each team to own and manage their
data independently, but the central data governance team needs to enforce consistent
data quality standards and make all datasets discoverable across the company. Which
architecture pattern and GCP services best support this?

**A)** Centralize all data into a single BigQuery project managed by the governance team
**B)** Implement a federated data mesh: each domain owns a Dataplex lake in their project; central Dataplex catalog with shared tag templates and quality rules
**C)** Use separate Cloud Storage buckets per team with individual IAM policies; manually catalog in a spreadsheet
**D)** Replicate all domain data to a central BigQuery dataset nightly for governance

**Answer: B**

> A federated data mesh with Dataplex enables domain autonomy (each team owns their
> lake) while the central team enforces governance via shared catalog, tag templates,
> and Dataplex data quality rules. Centralizing all data (A) creates a bottleneck;
> manual cataloging (C) doesn't scale; nightly replication (D) adds latency and cost.

### Question 2

A data platform team is building a unified data platform on GCP. The platform needs to:
(1) store raw data as files, (2) enable SQL analytics at scale, (3) manage metadata
and lineage, and (4) enforce column-level access control on Cloud Storage data. Which
combination of services forms the correct platform foundation?

**A)** Cloud Storage + Dataproc + Data Catalog (legacy) + Cloud SQL
**B)** Cloud Storage + BigQuery + Dataplex + BigLake
**C)** Cloud Storage + BigQuery + Cloud Monitoring + IAM
**D)** Cloud SQL + BigQuery + Pub/Sub + Cloud Composer

**Answer: B**

> Cloud Storage provides the data lake; BigQuery provides SQL analytics; Dataplex
> provides metadata, lineage, and governance; BigLake provides column-level security
> on Cloud Storage data via BigQuery. This is the current reference architecture for
> GCP data platforms.

### Question 3

A company uses Dataplex Catalog to manage metadata across their data platform. They
want business users to be able to tag datasets with business context (e.g., "data owner",
"sensitivity level", "business domain") and use these tags to filter search results.
How should the data engineer configure this?

**A)** Add business metadata as folder names in Cloud Storage buckets
**B)** Create BigQuery table descriptions and use BigQuery INFORMATION_SCHEMA for search
**C)** Define custom tag templates in Dataplex Catalog and create tags on assets using these templates
**D)** Use Cloud Monitoring custom metrics to track data asset metadata

**Answer: C**

> Dataplex Catalog tag templates allow defining structured business metadata schemas
> (e.g., fields for "data owner", "sensitivity level"). Tags created from templates
> are attached to data assets and are searchable/filterable in the Catalog UI and API.
> This is the idiomatic approach for business metadata management in Dataplex.

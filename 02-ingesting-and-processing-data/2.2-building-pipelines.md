## Key Concepts

> _Fill in your own notes here as you study._

- **Processing logic patterns**:
  - *Filtering*: Remove records that don't meet criteria
  - *Aggregation*: Group and summarize data (COUNT, SUM, AVG)
  - *Joining*: Combine data from multiple sources by key
  - *Enrichment*: Add new fields from reference data or ML models
  - *Deduplication*: Remove duplicate records (windowed dedup in streaming)
  - *Schema normalization*: Enforce consistent data types and field names

- **Dataflow transforms**: ParDo (element-wise), GroupByKey, Combine, CoGroupByKey,
  Flatten, Partition. Understanding PCollection and PTransform abstractions is key.

- **Windowing in streaming**: Fixed windows (tumbling), sliding windows, session windows.
  Critical for aggregating streaming data over time periods.

- **AI data enrichment**: Enriching records during pipeline execution with:
  - Pre-trained models (Vision AI, Natural Language AI, Translation AI)
  - Custom Vertex AI models called via REST/gRPC
  - BigQuery ML predictions inline with SQL
  - Embedding generation for unstructured data

- **Cloud Data Fusion transformations**: Wrangler (interactive data preparation),
  Joiner, Aggregator, Deduplicate, and custom plugins for code-based transforms.

## Practice Questions

### Question 1

A data pipeline receives customer support tickets as text via Pub/Sub. The pipeline must
classify each ticket's sentiment (positive/negative/neutral) and add this as a new field
before storing in BigQuery. The team wants minimal custom model development. Which
approach achieves this with the least operational overhead?

**A)** Train a custom TensorFlow sentiment model and deploy on Vertex AI
**B)** Use Dataflow with a DoFn that calls the Cloud Natural Language API for sentiment analysis
**C)** Load raw text into BigQuery and write a SQL function for sentiment detection
**D)** Use Cloud Data Fusion with a custom Java plugin for NLP

**Answer: B**

> Cloud Natural Language API is a pre-trained model requiring no custom ML development.
> A Dataflow DoFn can call the API per element within the streaming pipeline. This
> provides AI enrichment with minimal custom model work (vs. A) and is more suited to
> streaming than B's SQL-only approach.

### Question 2

A streaming Dataflow pipeline processes clickstream events and must compute a rolling
1-hour total count of page views per user. Events can arrive up to 10 minutes late.
Which Dataflow feature must the engineer configure?

**A)** Fixed windows of 1 hour with an allowed lateness of 10 minutes
**B)** Global windows with periodic triggers
**C)** Session windows with a 10-minute gap duration
**D)** Sliding windows of 10 minutes sliding every 1 hour

**Answer: A**

> Fixed windows divide the stream into non-overlapping 1-hour chunks for aggregation.
> Allowed lateness of 10 minutes tells Dataflow to accept and incorporate late-arriving
> data into the correct window rather than discarding it. Session windows (C) are
> for activity-based grouping, not fixed time intervals.

### Question 3

A data pipeline ingests product images from Cloud Storage and needs to extract text
(e.g., product labels) from each image and store it in BigQuery for search purposes.
The team wants a managed, serverless solution without running custom ML models. Which
architecture is most appropriate?

**A)** Use a Dataflow pipeline with a DoFn that calls the Cloud Vision API OCR feature per image
**B)** Deploy a custom OCR TensorFlow model on Vertex AI Endpoints
**C)** Use BigQuery ML with a remote model to process images
**D)** Write a Cloud Function triggered by Cloud Storage that calls the Translation API

**Answer: A**

> Cloud Vision API provides pre-trained OCR (text extraction from images). A Dataflow
> pipeline can process images at scale by calling Vision API per element in a ParDo.
> This is serverless, managed, and requires no custom model training. Option C is
> possible but BigQuery ML remote models are more suited for tabular or embedding tasks.

## Key Concepts

> _Fill in your own notes here as you study._

- **Cloud Composer vs. Workflows**:
  - *Cloud Composer*: Managed Apache Airflow. Best for complex DAGs, cross-service
    orchestration, Python-based logic, and teams familiar with Airflow. Higher cost.
  - *Workflows*: Fully serverless, pay-per-step. Best for simpler sequential or
    moderately conditional pipelines. Uses YAML/JSON syntax. Lower latency to start.

- **CI/CD for data pipelines**:
  - Source control: Store pipeline code in Cloud Source Repositories or GitHub
  - Build: Cloud Build compiles, tests, and packages pipeline artifacts
  - Test: Run unit tests on transform logic; integration tests against staging datasets
  - Deploy: Automated deployment of Dataflow templates, Dataform workspaces, or
    Composer DAG uploads to Cloud Storage
  - Promote: Separate dev/staging/prod environments with promotion gates

- **Dataflow Flex Templates**: Containerized pipeline templates stored in Artifact
  Registry, enabling versioned, repeatable deployments via CI/CD.

- **Dataform release configurations**: Version-controlled releases of transformation
  workspaces, supporting environment-specific variable injection.

- **Triggering patterns**: Event-driven (Cloud Storage trigger → Workflows),
  time-based (Cloud Scheduler → Composer DAG), and dependency-based (Composer sensors).

## Practice Questions

### Question 1

A data engineering team wants to automate the deployment of their Dataflow pipelines
whenever code is merged to the main branch in their GitHub repository. Pipeline updates
should be tested in a staging environment before being promoted to production. Which
GCP services should they use?

**A)** Cloud Scheduler to run the pipeline; Cloud Storage to store code
**B)** Cloud Build to trigger on GitHub push, run tests, build a Dataflow Flex Template, and deploy to staging/prod
**C)** Manually package the pipeline JAR and upload to Cloud Storage on each release
**D)** Cloud Composer to watch GitHub for changes and deploy pipelines

**Answer: B**

> Cloud Build integrates with GitHub via triggers, can run unit/integration tests,
> build a Flex Template container image, push to Artifact Registry, and deploy to
> staging/prod Dataflow jobs — a complete CI/CD workflow. Manual (C) is not automation.

### Question 2

A pipeline runs 12 sequential steps across BigQuery, Dataflow, and Cloud Storage, with
conditional logic (e.g., skip step 5 if step 4 produces zero rows). The pipeline must
complete within 30 minutes of triggering. Which orchestration tool is most appropriate?

**A)** Cloud Scheduler with separate Cloud Functions per step
**B)** Cloud Composer with a Python DAG using BranchPythonOperator for conditional logic
**C)** Workflows with a YAML-based step definition and conditional jumps
**D)** BigQuery scheduled queries chained via views

**Answer: B**

> Cloud Composer/Airflow's `BranchPythonOperator` handles conditional logic natively.
> For 12 steps with cross-service coordination and conditional branching, Composer is
> more appropriate than Workflows (C), which handles simpler conditional chains.
> Cloud Scheduler (A) has no built-in dependency management.

### Question 3

A team has separate GCP projects for development, staging, and production. Their
Dataform transformation workspace should use different BigQuery dataset names per
environment (e.g., `analytics_dev`, `analytics_prod`). How should they manage this?

**A)** Maintain three separate copies of all Dataform SQL files with hardcoded dataset names
**B)** Use Dataform compilation variables and release configurations to inject environment-specific dataset names
**C)** Use Cloud Build to find-and-replace dataset names in SQL files before deployment
**D)** Use separate GitHub branches for each environment with different SQL files

**Answer: B**

> Dataform supports compilation variables that can be set per release configuration.
> This allows one codebase to compile differently per environment without code
> duplication (A) or fragile string replacement (C). This is the idiomatic Dataform
> approach for multi-environment management.

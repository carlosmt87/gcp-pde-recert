## Key Concepts

> _Fill in your own notes here as you study._

- **ETL vs. ELT**: ETL (Extract-Transform-Load) transforms before loading — suits
  structured sources and when the warehouse can't handle raw data. ELT (Extract-Load-
  Transform) loads raw data first and transforms inside BigQuery using SQL/Dataform —
  preferred modern pattern with BigQuery.

- **Batch vs. streaming**: Batch pipelines process bounded datasets at scheduled
  intervals (daily, hourly). Streaming pipelines process unbounded data in near-real-time
  (seconds to minutes latency). Many architectures combine both (Lambda/Kappa).

- **Orchestration**: Coordination of multi-step pipelines across services. Determines
  execution order, handles dependencies, retries failures, and provides visibility.
  Cloud Composer (managed Airflow) is the primary GCP orchestration tool.

- **Transformation logic design**: Define what data transformations are needed (joins,
  aggregations, enrichments, deduplication) and select the right service based on:
  - Data volume and velocity
  - Latency requirements
  - Team skill set (SQL vs. code)
  - Managed vs. self-operated preference

- **DAG design**: Directed Acyclic Graphs define pipeline task dependencies.
  Well-designed DAGs are modular, idempotent, and observable.

## Practice Questions

### Question 1

A retail company receives point-of-sale transactions in real-time and needs to update
inventory counts within 5 minutes of each sale. They also need daily aggregated sales
reports in BigQuery. Which pipeline architecture should the data engineer design?

**A)** Batch pipeline running every 5 minutes for both use cases
**B)** Streaming pipeline for inventory updates; separate batch pipeline for daily reports
**C)** A single Dataform workflow that runs every 5 minutes
**D)** Manual exports from POS systems into Cloud Storage hourly

**Answer: B**

> Streaming (Pub/Sub → Dataflow → Bigtable/BigQuery) meets the 5-minute inventory SLA.
> A separate batch pipeline (e.g., scheduled Dataform job) handles daily aggregations
> efficiently. Running batches every 5 minutes (A) is inefficient and not true streaming.

---

### Question 2

A data engineer is designing a transformation pipeline that joins data from Cloud SQL
(operational database) with data already in BigQuery. The transformation involves complex
SQL logic and must run daily. Which approach is most operationally efficient?

**A)** Write a Python Dataflow job that connects to both Cloud SQL and BigQuery
**B)** Use BigQuery Federated Queries to read from Cloud SQL, then apply SQL transformations with Dataform
**C)** Export Cloud SQL data to CSV and import manually into BigQuery
**D)** Use Cloud Data Fusion with a drag-and-drop connector for both sources

**Answer: B**

> BigQuery Federated Queries allow direct SQL access to Cloud SQL, and Dataform manages
> the transformation logic as version-controlled SQL. This minimizes operational overhead
> (no Dataflow cluster management) for a SQL-centric team. D is valid but more complex
> to operate than pure SQL/Dataform for this use case.

### Question 3

An analytics team needs to orchestrate a pipeline that: (1) triggers a Dataflow job,
(2) waits for completion, (3) runs a BigQuery SQL transformation, (4) sends an email
notification. The pipeline has complex branching logic based on success/failure of each
step. Which tool is best suited?

**A)** Cloud Scheduler with individual Cloud Functions
**B)** Cloud Composer (Airflow) with a multi-task DAG
**C)** Workflows with a simple sequential step definition
**D)** BigQuery scheduled queries

**Answer: B**

> Cloud Composer/Airflow is purpose-built for complex multi-step DAGs with conditional
> branching, retry logic, and cross-service orchestration. Workflows (C) is better for
> simpler, linear chains without complex branching. Scheduler + Functions (A) lacks
> native dependency management.

## Key Concepts

> _Fill in your own notes here as you study._

- **BigQuery row-level security**: Row-level access policies control which users can
  see which rows in a table based on their identity. Uses row access policies linked
  to BigQuery tables. Filters data transparently at query time.

- **BigQuery column-level security**: Column-level access is implemented via BigQuery
  Policy Tags (managed in Data Catalog/Dataplex). Columns are tagged with policy tags;
  users without the `roles/datacatalog.categoryFineGrainedReader` role see NULLs.

- **BigQuery data masking**: Dynamic data masking applies masking rules (nullify, hash,
  default value) to columns based on the querying user's role, without duplicating data.
  Configured via Data Catalog masking rules on policy tag taxonomies.

- **Cloud DLP (Data Loss Prevention)**:
  - Detects sensitive data (PII, financial, health info) using 150+ built-in detectors
  - Can *inspect* (detect), *de-identify* (transform/mask), and *re-identify* data
  - De-identification methods: tokenization, masking, bucketing, date shifting, pseudonymization
  - Can be integrated into Dataflow pipelines for real-time de-identification
  - DLP inspection jobs can scan BigQuery tables, Cloud Storage, and Datastore

- **IAM for visualization**: Control access to BigQuery datasets and tables via IAM
  roles (`roles/bigquery.dataViewer`, `roles/bigquery.jobUser`). Looker Studio, Looker,
  and Connected Sheets inherit BigQuery permissions.

- **Authorized views**: A BigQuery feature to share query results (a view) with users
  who don't have access to the underlying tables. The view is "authorized" to query
  source tables on behalf of users.

## Practice Questions

### Question 1

A healthcare company stores patient data in BigQuery. Clinicians should see full patient
records including PII fields. Analysts should see all fields EXCEPT the `patient_name`
and `ssn` columns, which should appear as NULL. The solution must not require maintaining
separate tables. Which BigQuery feature achieves this?

**A)** Create separate BigQuery views for each user group with explicit column selection
**B)** Use BigQuery dynamic data masking with Policy Tags: apply a "nullify" masking rule on `patient_name` and `ssn`, grant clinicians `datacatalog.categoryFineGrainedReader`, grant analysts no fine-grained reader role
**C)** Use Cloud DLP to scan and remove PII from BigQuery tables nightly
**D)** Create authorized views that SELECT all columns except PII columns

**Answer: B**

> Dynamic data masking with Policy Tags provides column-level security where the masking
> behavior (nullify, hash, etc.) is applied at query time based on the user's IAM role.
> No data duplication. Views (A, D) require management overhead and don't dynamically
> adapt per user. Cloud DLP (C) modifies the underlying data, not access.

### Question 2

A company needs to process customer records containing credit card numbers stored in
Cloud Storage before loading to BigQuery. The pipeline must replace credit card numbers
with a format-preserving token so that records can be joined by card number across
systems, but the actual number is never stored in BigQuery. Which Cloud DLP de-identification
method should be used?

**A)** Masking with fixed-length characters (e.g., `****`)
**B)** Date shifting
**C)** Format-preserving encryption (FPE) tokenization
**D)** Bucketing (value generalization)

**Answer: C**

> Format-preserving encryption produces a token that maintains the original format (same
> length, digit structure) — meaning it can still be used as a join key across systems.
> Masking (A) destroys the join ability. Date shifting (B) is for dates. Bucketing (D)
> groups values into ranges, losing individual identity.

### Question 3

A Looker Studio dashboard queries a BigQuery table containing sales data across all
regions. Regional managers should only see data from their own region without the
dashboard developer needing to create separate dashboards per region. Which BigQuery
feature enables this?

**A)** Create one BigQuery view per region and connect each to a separate dashboard
**B)** Implement BigQuery row-level security with row access policies that filter rows by the querying user's region attribute
**C)** Use Cloud DLP to redact rows belonging to other regions
**D)** Use IAM conditions to filter BigQuery table access by region field value

**Answer: B**

> BigQuery row-level security creates row access policies that transparently filter
> query results based on the user's identity or group membership. A single dashboard
> connected to a single table automatically shows each manager only their region's data.
> Separate views/dashboards (A) create maintenance overhead; IAM conditions (D) operate
> at the resource level, not row level.

## Key Concepts

> _Fill in your own notes here as you study._

- **Feature engineering**: Transforming raw data into features that ML models can learn
  from. Includes:
  - Normalization / standardization (scaling numerical values)
  - One-hot encoding (converting categorical variables)
  - Bucketizing / binning (discretizing continuous values)
  - Feature crosses (combining features)
  - Handling missing values (imputation)
  - Time-based features (day of week, lag features)

- **BigQuery ML (BQML)**: Run ML model training and inference directly in BigQuery
  using SQL. Supports: linear/logistic regression, k-means, matrix factorization,
  XGBoost, TensorFlow, AutoML, and remote models (Vertex AI). No data movement needed.

- **Vertex AI Feature Store**: Centralized, managed repository for ML features.
  - Stores features with low-latency serving for online inference
  - Batch feature export for training
  - Ensures training-serving skew prevention (same features at train and serve time)

- **Training/serving patterns**:
  - *Batch prediction*: Run inference on large datasets offline (BigQuery ML, Vertex AI Batch Prediction)
  - *Online prediction*: Real-time inference on individual requests (Vertex AI Endpoints)
  - *Training-serving skew*: Using different feature transformations at train vs. serve time — a major source of ML bugs; Vertex AI Feature Store mitigates this

- **Embeddings**: Dense vector representations of data (text, images, code). Capture
  semantic meaning in a numerical form. Generated by embedding models (e.g., text-embedding-gecko
  via Vertex AI). Stored in vector databases or BigQuery VECTOR type for similarity search.

- **Retrieval-Augmented Generation (RAG)**: A pattern for grounding LLM responses in
  proprietary data:
  1. Convert documents to embeddings → store in vector store
  2. At query time, embed the user's question
  3. Retrieve top-k most similar document chunks via vector similarity search
  4. Include retrieved chunks as context in the LLM prompt
  GCP supports this via: Vertex AI, AlloyDB vector search, BigQuery vector search.

- **Unstructured data preparation for RAG**:
  - Extract text from PDFs, images (Vision AI OCR), audio (Speech-to-Text)
  - Chunk documents into appropriate segments
  - Generate embeddings for each chunk using Vertex AI embedding models
  - Store embeddings with metadata for retrieval

## Practice Questions

### Question 1

A data scientist wants to build a customer churn prediction model using data already
in BigQuery. The team has strong SQL skills but limited Python/TensorFlow experience.
The model needs to be trained monthly and predictions run in batch on all customers.
Which approach minimizes complexity while meeting requirements?

**A)** Export BigQuery data to CSV, train a scikit-learn model locally, and import predictions
**B)** Use BigQuery ML to train a logistic regression model with `CREATE MODEL`, and run batch predictions with `ML.PREDICT`
**C)** Build a Vertex AI Pipeline with a custom TensorFlow training job and Vertex AI Batch Prediction
**D)** Train an AutoML model in Vertex AI and export predictions to BigQuery

**Answer: B**

> BigQuery ML is purpose-built for this scenario: SQL-skilled team, data already in
> BigQuery, batch monthly predictions. `CREATE MODEL` trains the model; `ML.PREDICT`
> runs batch inference — all in SQL, no data movement, no infrastructure management.
> Option C and D are valid but significantly more complex for a SQL team.

### Question 2

A company is building a RAG-based chatbot that answers questions about their internal
policy documents (stored as PDFs in Cloud Storage). They need to implement the vector
retrieval component. Which sequence of steps correctly prepares the data for RAG?

**A)** Store PDFs in BigQuery BLOB columns → query PDFs directly with SQL → pass to LLM
**B)** Extract text from PDFs using Vision AI OCR → chunk text → generate embeddings via Vertex AI Embeddings API → store embeddings in Vertex AI Vector Search index → at query time, embed question and retrieve top-k chunks → pass to Gemini
**C)** Convert PDFs to images → store in Cloud Storage → use Gemini Vision to read PDFs directly at query time
**D)** Index PDFs in Cloud Search → use Cloud Search API for retrieval → pass results to LLM

**Answer: B**

> This is the correct RAG pipeline: extract → chunk → embed → index → retrieve → generate.
> Option B correctly uses: Vision AI for text extraction, Vertex AI Embeddings for
> vector generation, Vector Search for similarity retrieval, and Gemini for generation.
> Option C skips the indexing step, making it expensive and slow at query time.

### Question 3

A data engineering team is building a feature pipeline for a real-time fraud detection
model. Features computed at training time (e.g., 30-day rolling average transaction
amount) must be identical to features computed at inference time. What is the most
important architectural consideration and which GCP service addresses it?

**A)** Use the same Python version for training and inference; deploy both on Compute Engine
**B)** Use Vertex AI Feature Store to define, store, and serve features consistently for both training batch export and online inference, preventing training-serving skew
**C)** Write features to a BigQuery table and query it from both the training job and the inference service
**D)** Use Cloud Memorystore to cache computed features for the inference service

**Answer: B**

> Training-serving skew (different feature values at train vs. serve time) is a critical
> ML reliability issue. Vertex AI Feature Store provides a single source of truth for
> features: batch export for training, low-latency online serving for inference — using
> the same feature definitions, ensuring consistency. BigQuery (C) works for offline
> but isn't designed for low-latency online serving.

## Key Concepts

> _Fill in your own notes here as you study._

- **Dataform**: A SQL-based data transformation framework integrated into BigQuery.
  Manages dbt-style transformations, tests, and schedules. Supports version control
  and incremental tables. Best for ELT patterns in BigQuery.

- **Dataflow**: Managed Apache Beam service for stream and batch data processing.
  Handles windowing, triggers, and stateful processing for reliable pipeline execution.
  Supports exactly-once processing semantics.

- **Cloud Data Fusion**: Fully managed, no-code/low-code ETL service built on CDAP.
  Provides a visual pipeline designer. Good for teams needing code-free data integration.

- **LLM-assisted query generation**: Using Gemini or other LLMs (via Vertex AI) to
  generate SQL queries or transformation logic from natural language prompts. Reduces
  manual coding errors and accelerates pipeline development.

- **Data validation strategies**: Schema validation, referential integrity checks,
  null/duplicate detection, statistical profiling, and row count reconciliation.

- **Idempotency**: Pipelines should produce the same output if run multiple times with
  the same input — critical for reliable reprocessing after failures.

## Practice Questions

### Question 1

A data engineering team uses BigQuery as their data warehouse. They need a solution to
manage SQL-based data transformations, enforce data quality tests, and track lineage
— all within BigQuery, with version-controlled code. Which service best fits?

**A)** Cloud Data Fusion
**B)** Dataflow
**C)** Dataform
**D)** Cloud Composer

**Answer: C**

> Dataform is purpose-built for SQL-based ELT within BigQuery. It supports dbt-style
> assertions (data quality tests), incremental models, and Git integration. Cloud Data
> Fusion (A) is a visual ETL tool; Dataflow (B) uses Java/Python, not SQL-native.

---

### Question 2

A streaming pipeline reads from Pub/Sub and writes to BigQuery. During a downstream
outage, messages accumulate in Pub/Sub. When the pipeline resumes, the team wants to
ensure no duplicate records appear in BigQuery. Which Dataflow capability addresses this?

**A)** Autoscaling worker nodes
**B)** Exactly-once processing semantics
**C)** Side inputs from Cloud Storage
**D)** Windowing with fixed intervals

**Answer: B**

> Dataflow's exactly-once processing guarantees that each message is processed and
> written to BigQuery once, even when the pipeline restarts or retries. Autoscaling (A)
> is about capacity, not correctness.

---

### Question 3

A data engineer is building a pipeline to ingest customer records from a CRM system
into BigQuery. Before loading, they need to validate that: (1) no null values exist in
the `customer_id` column, (2) email addresses match a regex pattern, and (3) record
counts match the source. Which approach is most appropriate for ongoing automated
validation?

**A)** Manually run SQL queries in BigQuery after each load
**B)** Use Dataform assertions to define and run these checks as part of the transformation DAG
**C)** Write custom Python scripts and schedule them with cron
**D)** Use Cloud DLP to scan for sensitive fields

**Answer: B**

> Dataform assertions let you define reusable data quality checks (not-null, regex,
> row count) that run automatically as part of the transformation pipeline. This provides
> systematic, version-controlled validation — not manual (A) or ad-hoc (C) approaches.

## Key Concepts

> _Fill in your own notes here as you study._

- **BigQuery pricing models**:
  - *On-demand*: Pay per TB of data scanned. Good for unpredictable, sporadic workloads.
    Default for new users. No capacity commitment.
  - *Capacity-based (Editions)*: Purchase compute slots; unlimited queries within
    capacity. Predictable costs for consistent workloads.

- **BigQuery Editions** (the modern capacity model):
  - *Standard Edition*: Lowest cost per slot-hour; suitable for batch and less
    time-sensitive workloads. Autoscaling with maximum slot cap.
  - *Enterprise Edition*: Adds features like CMEK, BI Engine, and higher SLAs.
    Better for production analytical workloads.
  - *Enterprise Plus Edition*: Highest tier; includes features like time travel up
    to 7 days, table clones, and reserved slots with guaranteed capacity.

- **Slots**: Unit of BigQuery compute. One slot = one virtual CPU for query execution.
  Complex queries use more slots in parallel. Default on-demand accounts get a shared
  slot pool with fair scheduling.

- **Reservations**: Assign a fixed number of slots to a specific project, folder, or
  organization. Ensures that workloads get guaranteed capacity regardless of other
  queries running in the organization.
  - *Baseline slots*: Always-on committed slots
  - *Autoscaling slots*: Burst capacity on top of baseline (billed per slot-second used)
  - *Idle slots sharing*: Unused slots from one reservation can be shared to others

- **Assignments**: Map a project (or folder/org) to a specific reservation. A project
  without an assignment uses on-demand pricing.

- **Workload separation**: Use separate reservations for different workload types
  (e.g., one reservation for interactive BI, another for batch ETL jobs). Prevents
  heavy batch jobs from starving interactive dashboards of slots.

- **Commitments**: 1-year or 3-year slot commitments reduce the per-slot-hour cost
  compared to on-demand autoscaling slots.

## Practice Questions

### Question 1

A company has two BigQuery workloads: (1) a BI dashboard queried by executives 9am-5pm
on business days, requiring consistently fast response times; (2) a nightly batch ETL
pipeline that runs 11pm-3am and can tolerate longer runtimes. The company currently
uses on-demand pricing and experiences dashboard slowdowns during busy periods. Which
capacity strategy resolves this?

**A)** Switch all workloads to on-demand and add query labels for prioritization
**B)** Purchase an Enterprise Edition commitment; create separate reservations for BI and ETL workloads; assign each project to its respective reservation; configure idle slot sharing from ETL to BI during business hours
**C)** Schedule the ETL pipeline to run only at weekends to avoid conflict
**D)** Use BigQuery BI Engine for the dashboard and on-demand for ETL

**Answer: B**

> Separate reservations guarantee that neither workload starves the other. Idle slot
> sharing allows ETL's unused slots (during business hours) to flow to the BI reservation,
> maximizing efficiency. BI Engine (D) helps with in-memory caching but doesn't solve
> slot contention. Scheduling ETL to weekends (C) is an artificial operational constraint.

### Question 2

A data platform team manages BigQuery for a large organization with 50 teams across 3
business units. They want to ensure that a single team's runaway query cannot consume
all available slots and degrade performance for all other teams. Which BigQuery feature
should be configured?

**A)** Enable BigQuery query cost controls at the project level (maximum bytes billed)
**B)** Create reservations for each business unit and assign all projects to their respective
   unit's reservation with autoscaling maximums set per unit
**C)** Use column-level security to limit which tables teams can query
**D)** Set per-user query quotas using IAM conditions

**Answer: B**

> Reservations with autoscaling maximums cap how many slots each business unit can
> consume. A runaway query in one unit can only use that unit's allocated slots and
> cannot overflow into other units' reservations. Maximum bytes billed (A) limits query
> cost but not slot consumption rate. IAM conditions (D) don't control slot usage.

### Question 3

A startup uses BigQuery on-demand pricing. Their data engineering team runs large
exploratory queries that scan terabytes of data, generating unexpected high costs. At
the same time, their BI tool runs frequent, fast queries. Which approach best controls
costs while maintaining BI performance?

**A)** Delete old partitions frequently to reduce data scanned
**B)** Switch to BigQuery Enterprise Edition with a reservation; set a slot autoscaling maximum to cap cost; assign the project to the reservation
**C)** Require all queries to use `SELECT *` to avoid partial scans
**D)** Move the BI tool to Cloud SQL to avoid BigQuery costs

**Answer: B**

> Capacity-based pricing with a slot autoscaling maximum converts unpredictable per-TB
> costs into a bounded slot-hour cost. Both exploratory and BI queries share the reserved
> capacity without per-query cost spikes. Deleting data (A) destroys analytical value;
> `SELECT *` (C) is wrong — it increases scanned data; moving to Cloud SQL (D) loses
> BigQuery's analytical capabilities.

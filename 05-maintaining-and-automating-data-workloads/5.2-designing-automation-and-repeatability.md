## Key Concepts

> _Fill in your own notes here as you study._

- **Idempotency**: A pipeline run produces the same result regardless of how many times
  it's executed with the same input. Critical for safe retries and backfills.
  - Use `MERGE` instead of `INSERT` in BigQuery to prevent duplicates
  - Use write dispositions: `WRITE_TRUNCATE` (replace) vs. `WRITE_APPEND` (accumulate)
  - Partition-based overwrite: Overwrite specific date partitions, not entire tables

- **Backfilling**: Re-running a pipeline for historical time periods (e.g., after a
  bug fix). Well-designed idempotent pipelines support backfilling safely. Cloud
  Composer DAGs with `start_date`, `end_date`, and catchup settings manage backfills.

- **Scheduling strategies**:
  - *Time-based*: Cloud Scheduler, Composer `schedule_interval`, BigQuery Scheduled Queries
  - *Event-driven*: Cloud Storage triggers via Eventarc or Pub/Sub → Workflows/Composer
  - *Dependency-based*: Upstream job completion triggers downstream (Composer sensors,
    Dataform triggers, Workflows callbacks)

- **Parameterized pipelines**: Design pipelines to accept input parameters (date range,
  dataset name, batch size) rather than hardcoding values. Enables reuse and repeatability
  across environments and time periods.

- **Dataform schedules**: Native scheduling within Dataform workspaces; trigger
  workflow invocations on a cron schedule with compilation variables injected per run.

- **BigQuery Scheduled Queries**: Simple recurring SQL queries scheduled within
  BigQuery. Good for lightweight aggregations and reporting tables. Not suitable for
  complex multi-step orchestration.

## Practice Questions

### Question 1

A data pipeline appends daily sales data to a BigQuery table. A bug in the pipeline
caused duplicate records to be inserted for 3 days. After fixing the bug, the team
needs to re-run the pipeline for the affected days without creating further duplicates.
Which BigQuery write pattern should be used for the pipeline to support safe reprocessing?

**A)** Use `WRITE_APPEND` to add corrected records alongside existing duplicates
**B)** Delete all data from the table and rerun the full pipeline from the beginning
**C)** Use date partitioning with `WRITE_TRUNCATE` partition override: overwrite each affected date partition with corrected data
**D)** Manually delete duplicate rows using a SQL `DELETE` statement then append corrected data

**Answer: C**

> Partitioned tables with `WRITE_TRUNCATE` at the partition level replace only the
> affected date partition(s), not the entire table. This is the idiomatic idempotent
> pattern for daily pipelines in BigQuery — safe to re-run for any date without
> affecting other partitions or accumulating duplicates.

### Question 2

A Dataform pipeline runs every hour and transforms new data in BigQuery. Occasionally,
the upstream Dataflow job that populates the source table runs late, causing Dataform
to process incomplete data. How should the pipeline be designed to handle this?

**A)** Run Dataform every 15 minutes to reduce the chance of incomplete source data
**B)** Use Cloud Composer to orchestrate: a sensor checks that the Dataflow job completed successfully before triggering the Dataform workflow invocation
**C)** Add a BigQuery Scheduled Query to validate row counts before Dataform runs
**D)** Configure Dataform to retry failed transformations 3 times

**Answer: B**

> Cloud Composer with a sensor (e.g., `DataflowJobStateSensor`) checks for upstream
> job completion before triggering Dataform. This creates a dependency-based scheduling
> model rather than time-based, ensuring Dataform only runs on complete source data.
> Shorter intervals (A) are wasteful; retries (D) don't solve the root cause.

### Question 3

A data pipeline needs to process data for each day of the previous week every Monday.
The pipeline is implemented as a Cloud Composer DAG. The team wants to run it manually
for any past week during an incident. Which Composer/Airflow features support this?

**A)** Trigger the DAG manually in the Airflow UI with a hardcoded date parameter
**B)** Design the DAG with `schedule_interval='@weekly'`, `start_date` set historically, `catchup=True`, and parameterize date logic using `{{ ds }}` Jinja macros; trigger backfills via `airflow dags backfill` CLI
**C)** Create a separate "backfill DAG" with hardcoded date ranges for incident recovery
**D)** Use Cloud Scheduler to trigger the DAG repeatedly with different dates

**Answer: B**

> Airflow's `catchup=True` and Jinja templating (`{{ ds }}`, `{{ data_interval_start }}`)
> make DAGs inherently parameterized by execution date. The `airflow dags backfill`
> command triggers historical runs safely. Separate backfill DAGs (C) are maintenance
> overhead; hardcoded params (A) require manual code changes per incident.

## Key Concepts

> _Fill in your own notes here as you study._

- **Observability pillars**: Metrics (what happened), Logs (why it happened), Traces
  (where it happened). GCP provides all three via Cloud Monitoring, Cloud Logging,
  and Cloud Trace.

- **Cloud Monitoring for data pipelines**:
  - Pre-built dashboards for Dataflow, Pub/Sub, BigQuery
  - Custom metrics from Dataflow user metrics (using `Metrics.counter()`, etc.)
  - Alerting policies on metric thresholds (e.g., Dataflow worker error rate,
    Pub/Sub subscription backlog age, BigQuery slot utilization)
  - Uptime checks for pipeline health endpoints

- **Cloud Logging for pipelines**:
  - Dataflow worker logs (structured logging via the Beam logging API)
  - BigQuery audit logs: `cloudaudit.googleapis.com/data_access` for query history
  - Cloud Composer/Airflow logs: DAG run logs, task logs in Cloud Logging
  - Log-based metrics: Create custom metrics from log patterns (e.g., count of
    error log entries per pipeline)
  - Log sinks: Export logs to BigQuery for long-term analysis

- **BigQuery Admin Panel**: BigQuery's built-in observability tool (in the GCP Console).
  Shows:
  - Job history (all queries, load, export, copy jobs) with execution details
  - Slot utilization charts per reservation
  - Table access patterns
  - INFORMATION_SCHEMA system views for programmatic access to the same data

- **INFORMATION_SCHEMA views**: SQL-queryable metadata in BigQuery.
  - `INFORMATION_SCHEMA.JOBS_BY_*`: Query history, bytes processed, slot ms, errors
  - `INFORMATION_SCHEMA.TABLE_STORAGE`: Table sizes and partition stats
  - `INFORMATION_SCHEMA.RESERVATIONS`: Slot reservation details
  - Useful for building custom monitoring dashboards and SLA reporting

- **Dataflow monitoring**: Dataflow provides a UI showing job graph, step-level metrics
  (elements processed, backlog, wall time), and system/worker logs. Key metrics:
  - `system_lag` (streaming): Time between event creation and processing
  - `backlog_bytes`/`backlog_elements`: Unprocessed data volume in source
  - Worker CPU/memory utilization

- **Pub/Sub monitoring**: Key metrics for streaming pipeline health:
  - `subscription/oldest_unacked_message_age`: Measures consumer lag
  - `topic/send_message_count`: Publisher throughput
  - Alert when oldest unacked age exceeds SLA threshold

- **Data freshness monitoring**: Use Cloud Monitoring custom metrics or BigQuery
  Scheduled Queries to check that tables were updated within expected timeframes.
  Alert when `max(ingestion_timestamp) < NOW() - interval` threshold is breached.

## Google Cloud Services

| Service | Role |
|---------|------|
| **Cloud Monitoring** | Metrics, dashboards, alerting for all GCP data services |
| **Cloud Logging** | Structured logs from Dataflow, Composer, BigQuery, Pub/Sub |
| **Cloud Trace** | Distributed tracing for pipeline latency analysis |
| **BigQuery Admin Panel** | Built-in job history, slot utilization, and query performance |
| **BigQuery INFORMATION_SCHEMA** | SQL-based programmatic access to metadata and job history |
| **Dataflow UI** | Job graph visualization, step metrics, and worker logs |
| **Error Reporting** | Aggregates and surfaces exceptions from pipeline logs |
| **Cloud Alerting** | Threshold-based and metric-absence alerts with notification channels |

## Practice Questions

### Question 1

A Dataflow streaming pipeline reads from Pub/Sub and writes to BigQuery. Users report
that data in the BigQuery dashboard appears increasingly delayed. The data engineer
needs to identify the bottleneck. Which metric and tool combination should they check
first?

**A)** Check Cloud Monitoring for BigQuery `table/row_count` metric
**B)** Check Dataflow Job monitoring for `system_lag` (time between event creation and processing) and `backlog_elements` in the Pub/Sub subscription; check worker CPU utilization
**C)** Query BigQuery `INFORMATION_SCHEMA.JOBS_BY_PROJECT` for slow queries
**D)** Check Cloud Logging for BigQuery Data Access audit logs

**Answer: B**

> `system_lag` directly measures the delay between event time and processing time —
> the metric that corresponds to user-reported delays. `backlog_elements` shows if the
> pipeline is falling behind the source. CPU utilization indicates if workers need to
> scale. BigQuery INFORMATION_SCHEMA (C) would show query performance, not pipeline lag.

### Question 2

A data engineering team needs to identify all BigQuery queries that scanned more than
100 GB in the last 24 hours, along with the user who ran them and the total bytes billed.
Which approach is most efficient?

**A)** Export BigQuery Admin Panel job history to a CSV and filter in a spreadsheet
**B)** Write a Cloud Monitoring alerting policy for the bytes scanned metric
**C)** Query `INFORMATION_SCHEMA.JOBS_BY_PROJECT` filtering on `total_bytes_processed > 100 * 1024^3` and `creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)`
**D)** Enable BigQuery Data Access audit logs and search Cloud Logging manually

**Answer: C**

> `INFORMATION_SCHEMA.JOBS_BY_PROJECT` provides structured, queryable job history
> including `user_email`, `total_bytes_processed`, `total_bytes_billed`, and
> `creation_time`. A targeted SQL query returns exactly the needed data programmatically.
> Manually searching logs (D) or CSVs (A) are error-prone and slow.

### Question 3

A Cloud Composer DAG has been running successfully for months. After a new operator was
added to the DAG, daily runs started failing at task 3 of 8. The data engineer needs
to diagnose the root cause. Which troubleshooting approach is most effective?

**A)** Re-deploy the DAG and wait for the next scheduled run to see if it self-heals
**B)** In the Airflow UI, click on the failed task, view its logs; cross-reference with Cloud Logging using the task's `run_id` to find full error stack traces and upstream resource logs
**C)** Check Cloud Monitoring for Composer environment CPU utilization
**D)** Query BigQuery INFORMATION_SCHEMA to find slow queries triggered by the DAG

**Answer: B**

> The Airflow UI task logs show the direct error from the failing task. Cross-referencing
> with Cloud Logging (filtering by `run_id` or task labels) provides the complete stack
> trace and any errors from downstream services the task called (e.g., Dataflow, BigQuery).
> CPU monitoring (C) and BigQuery logs (D) are unlikely to reveal the root cause of a
> DAG task failure.
